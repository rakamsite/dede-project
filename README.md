# پروژه DeDe

این مخزن برای مدیریت و پیگیری قالب و افزونه‌های اختصاصی DeDe در WordPress/WooCommerce استفاده می‌شود.

## محدوده پروژه

- `wp-content/themes/DeDeTemPlate`
- `wp-content/plugins/DeDeV1`
- `wp-content/plugins/DeDeV2`
- `wp-content/plugins/dede-core`
- `wp-content/plugins/dede-store-features`
- `docs/`
- `tools/`

فایل‌های هسته WordPress، پوشه `uploads`، کش، لاگ‌ها، بکاپ‌ها، `node_modules`، `vendor` و فایل‌های نامرتبط جزو Git این پروژه نیستند.

# گزارش انجام کار

## فاز 0 — آماده‌سازی Git و ساختار پروژه

### هدف / اثر این فاز

این فاز پروژه را از تغییرات پراکنده و ثبت‌نشده به یک روند کنترل‌شده مبتنی بر Git منتقل کرد تا تغییرات بعدی قابل پیگیری، قابل بررسی، امن‌تر و مناسب‌تر برای ثبت گزارش کاری باشند.

### موارد انجام‌شده

0.1. محدوده فایل‌های اختصاصی پروژه مشخص شد.  
0.2. قالب `DeDeTemPlate` به‌عنوان قالب اصلی پروژه در ساختار Git قرار گرفت.  
0.3. افزونه‌های اختصاصی `DeDeV1` و `DeDeV2` به‌عنوان محدوده اصلی توسعه مشخص شدند.  
0.4. فایل‌ها و پوشه‌های غیرضروری مثل آپلودها، کش، لاگ‌ها، بکاپ‌ها، `node_modules` و فایل‌های حجیم از محدوده Git و خروجی بررسی کنار گذاشته شدند.  
0.5. ساختار اولیه پوشه‌های `docs` و `tools` برای نگهداری مستندات داخلی و ابزارهای پروژه آماده شد.  
0.6. مسیر خروجی‌گیری تمیز از پروژه برای بررسی مرحله‌ای قالب، افزونه‌ها و مستندات آماده شد.  
0.7. مرجع واحد قوانین پروژه و ساختار اسناد داخلی با `AGENTS.md` و `docs/PROJECT-MASTER.md` یکپارچه شد.

## فاز 1 — پایداری لود قالب و افزونه‌ها

### هدف / اثر این فاز

این فاز برای کاهش احتمال خطاهای کشنده، صفحه سفید و وابستگی‌های ناامن در شروع لود قالب و افزونه‌ها انجام شد تا در نبود برخی فایل‌ها یا وابستگی‌ها، سایت با ریسک کمتری بالا بیاید.

### موارد انجام‌شده

1.1. لود اولیه قالب از وابستگی‌های دستی و ناامن به فایل‌های WooCommerce و WordPress پاک‌سازی شد و لود فایل‌های داخلی با مسیرهای امن‌تر انجام شد.  
1.2. برای فایل‌های داخلی قالب و فایل `assets/cmb2/init.php` شرط وجود فایل اضافه شد تا در صورت نبودن فایل، خطای کشنده ایجاد نشود.  
1.3. در افزونه‌های `DeDeV1` و `DeDeV2` لود `vendor/autoload.php` و شروع اجرای پلاگین با شرط‌های ایمن‌تر انجام شد تا نبود autoload باعث fatal error نشود.  
1.4. زیرساخت اولیه افزونه مرکزی برای مدیریت وابستگی‌ها و سرویس‌های مشترک ایجاد شد.  
1.5. سرویس مشترک و کنترل‌شده ثبت خطا برای توسعه قابلیت‌های بعدی ایجاد شد.

## فاز 4 — بازسازی اطلاعات مشتری

### هدف / اثر این فاز

فرایند انتخاب نوع حساب و تکمیل اطلاعات مشتری از منطق پراکنده و ناامن قالب به یک افزونه مستقل منتقل شد تا تجربه کاربری موبایل و دسکتاپ ساده‌تر، اعتبارسنجی اطلاعات دقیق‌تر و اتصال موجود شایگان بدون تغییر قرارداد داده حفظ شود.

### موارد انجام‌شده

4.1. افزونه `dede-store-features` برای مدیریت نوع حساب، پروفایل مشتری و کنترل اطلاعات Checkout ایجاد شد.  
4.2. فرم اطلاعات مشتری با مسیر سه‌مرحله‌ای مشخصات، آدرس و تأیید برای موبایل و دسکتاپ بازطراحی شد.  
4.3. متاکی‌های فعلی WooCommerce و شایگان بدون تغییر نام حفظ و اعتبارسنجی سمت سرور به آن‌ها اضافه شد.  
4.4. انتخاب نوع حساب `شخص`، `فروشگاه` و `شرکت` امن شد و کاربران دارای مرحله نیمه‌کاره در ورود بعدی به ادامه انتخاب هدایت می‌شوند.  
4.5. ثبت سفارش با اطلاعات ناقص مسدود و snapshot اطلاعات مشتری روی سفارش ذخیره شد.  
4.6. پل سازگاری قالب و fallback فرم قدیمی برای فعال‌سازی و rollback کنترل‌شده اضافه شد.
