# سند مادر توسعه DeDe

## 1. هدف پروژه

هدف این پروژه بازنویسی کامل Laravel نیست. هدف، **بازسازی تدریجی و اصولی در اکوسیستم WordPress/WooCommerce** است؛ به شکلی که سایت فعلی نخوابد، فروشگاه و سفارش‌ها حفظ شوند، اما به مرور از حالت قاطی‌پاتی و وصله‌پینه خارج شود.

قانون اصلی:

> WooCommerce موتور فروشگاه باقی می‌ماند. منطق تجاری اختصاصی از قالب خارج می‌شود. قالب فقط نقش پوسته و نمایش را نگه می‌دارد.

## 2. وضعیت فعلی شناخته‌شده

پروژه فعلی شامل یک قالب و دو افزونه است:

1. قالب `DeDeTemPlate`
2. افزونه `DeDeV1`
3. افزونه `DeDeV2`

قالب فعلی فقط ظاهر نیست و منطق‌های مهمی داخل آن قرار گرفته‌اند؛ از جمله WooCommerce overrideها، AJAXهای متعدد، ورود/ثبت‌نام پیامکی، پنل حساب کاربری، کیف پول، گارانتی، لیست قیمت و موجودی، ویدئو/پلی‌لیست، مگامنو، تنظیمات CMB2 و بخشی از قرارداد اتصال با حسابگر/شایگان.

بنابراین تغییرات نباید بدون نقشه انجام شوند.

## 3. اصل اجرایی فازها

هر فاز باید دو خروجی داشته باشد:

### 3.1 خروجی عملیاتی

یعنی قابلیت واقعی سایت پایدارتر شود؛ مثل رفع خطا، درست شدن مسیر خرید، بهبود ورود پیامکی، اصلاح پنل، یا پایدار شدن import محصول.

### 3.2 خروجی معماری

یعنی همان بخشی که لمس شده، تا حد ممکن به محل درست منتقل شود؛ مثلاً منطق OTP از قالب به افزونه، یا منطق import از قالب/افزونه legacy به ماژول Product Operations.

قانون:

> هیچ فازی نباید فقط وصله‌پینه باشد. هر فاز باید حداقل بخشی از بدهی فنی همان محدوده را کم کند.

## 4. فازهای اصلی

### فاز 0 — آماده‌سازی، Git، Backup و Staging

هدف: ایجاد محیط امن برای تغییرات.

خروجی‌های لازم:

- قرار گرفتن پروژه در Git
- بکاپ کامل فایل‌ها و دیتابیس
- ساخت محیط staging
- ثبت نسخه WordPress، WooCommerce، PHP، قالب و افزونه‌ها
- فعال‌سازی لاگ خطا
- تعریف branch اصلی و branch توسعه
- تست مسیرهای اصلی قبل از هر تغییر

این فاز پیش‌نیاز همه فازهاست.

### فاز 1 — Core، Guard و وابستگی‌ها

هدف: جلوگیری از صفحه سفید و خطاهای کشنده.

محدوده:

- Guard برای WooCommerce
- Guard برای CMB2
- لود امن فایل‌های قالب و افزونه
- جلوگیری از fatal error در محیط ناقص
- آماده‌سازی سرویس‌های مشترک مثل SMS، تاریخ شمسی، استان/شهر و logger
- کاهش ریسک `functions.php`

خروجی معماری:

- شروع ساخت یا آماده‌سازی `dede-core`
- انتقال helperها و سرویس‌های مشترک از قالب به Core، در صورت کم‌ریسک بودن

### فاز 2 — عملیات محصول، Excel، PDF و لیست قیمت

هدف: پایدار کردن قلب دیتای فروشگاه.

محدوده:

- import محصولات مادر
- import variationها
- بروزرسانی بر اساس SKU
- ساخت دسته‌بندی و attribute
- تصویر شاخص و گالری
- توضیحات محصول
- متاهای حسابگر/شایگان
- batch، pause/resume و گزارش خطا
- خروجی PDF
- لیست قیمت و موجودی

خروجی معماری:

- انتقال تدریجی عملیات محصول به `dede-product-operations`
- جدا شدن import/export از قالب و منطق front-end

### فاز 3 — مسیر خرید و WooCommerce حیاتی

هدف: پایدار شدن مسیر از محصول تا سفارش.

محدوده:

- single product
- archive محصول
- add to cart AJAX
- minicart
- cart
- checkout
- variation price
- order status
- noticeها
- تماس با شرکت
- لغو سفارش pending

خروجی معماری:

- کاهش overrideهای غیرضروری WooCommerce
- انتقال منطق تجاری سبد/checkout به افزونه، نه قالب
- نگه داشتن templateهای نمایشی در قالب

### فاز 4 — ورود، ثبت‌نام، پنل مشتری و اطلاعات کاربر

هدف: پایدار شدن تجربه کاربر و خروج منطق حساب کاربری از قالب.

محدوده:

- ورود و ثبت‌نام اختصاصی
- OTP
- فراموشی رمز با پیامک
- نقش و نوع کاربری
- پروفایل حقیقی/حقوقی
- آدرس‌ها
- استان/شهر
- سفارش‌های من
- جزئیات سفارش در پنل

خروجی معماری:

- انتقال OTP، profile، account logic به `dede-store-features`
- قالب فقط فرم‌ها و UI را نگه دارد

### فاز 5 — کیف پول، گارانتی و پیامک‌های فروش

هدف: پایدار کردن قابلیت‌های مالی/خدماتی حساس.

محدوده:

- کیف پول
- سقف شارژ و سقف مصرف
- سفارش شارژ کیف پول
- پرداخت با کیف پول
- گارانتی
- مدیریت گارانتی
- سبد خرید رهاشده
- پیامک سبد رهاشده
- پیامک تکمیل سفارش
- موجود شد خبرم کن

خروجی معماری:

- انتقال wallet، warranty و notifications به `dede-store-features`
- استفاده از سرویس پیامک مرکزی در `dede-core`

### فاز 6 — جستجو، منو، دسته‌بندی و ظاهر فروشگاه

هدف: مرتب شدن بخش‌های نمایشی و فروشگاهی کم‌ریسک‌تر.

محدوده:

- AJAX search
- quick view
- mega menu
- mobile menu
- category banner/meta
- slider
- popup
- banner فروش ویژه
- footer
- 404
- search page
- notice UI
- CSS/JS enqueue

خروجی معماری:

- نگه داشتن UI در قالب
- انتقال منطق داده‌ای یا AJAXهای غیرنمایشی به افزونه مناسب

### فاز 7 — محتوا، ویدئو، استوری و مجله

هدف: تصمیم‌گیری درباره امکانات محتوایی و حذف یا تمیزکاری موارد غیرضروری.

محدوده:

- ویدئو
- استوری ویدئو
- کاور
- لایک
- کامنت
- CTA
- محصولات مرتبط با ویدئو
- playlist
- single video
- صفحات مجله

خروجی معماری:

- اگر این امکانات استفاده واقعی دارند، به ماژول محتوایی منتقل شوند
- اگر استفاده واقعی ندارند، غیرفعال یا حذف تدریجی شوند

### فاز 8 — پاکسازی نهایی و خروج از وصله‌پینه

هدف: تمیز شدن نهایی مالکیت کدها.

محدوده:

- سبک شدن `functions.php`
- حذف کدهای مرده
- حذف duplicateها
- کم کردن WooCommerce overrideها
- استانداردسازی AJAXها
- مرتب‌سازی ساختار افزونه‌ها
- مستندسازی نهایی مالکیت قابلیت‌ها

خروجی نهایی:

- قالب فقط ظاهر و templateها را نگه دارد
- منطق تجاری داخل افزونه‌ها باشد
- قابلیت‌ها قابل تست، قابل گزارش و قابل قیمت‌گذاری باشند

## 6. Definition of Done برای هر فاز

هر فاز فقط وقتی تمام‌شده محسوب می‌شود که این موارد تکمیل شده باشد:

- کد در branch مشخص انجام شده باشد
- فایل‌های تغییرکرده لیست شده باشند
- مسیرهای اصلی تست شده باشند
- خطای fatal جدید ایجاد نشده باشد
- فارسی و UTF-8 خراب نشده باشد
- گزارش تغییرات در `CLIENT_PROGRESS_LOG.md` ثبت شده باشد
- موارد قابل ارائه به کارفرما مشخص شده باشد
- موارد باقی‌مانده و ریسک‌ها نوشته شده باشد

## 7. اصل مالی و ارائه به کارفرما

برای دریافت هزینه، هر تغییر باید قابل گزارش باشد. گزارش باید نشان دهد:

- چه مشکلی وجود داشت
- چه کاری انجام شد
- چه فایل‌ها/بخش‌هایی تغییر کرد
- نتیجه قابل مشاهده چیست
- چطور تست شد
- چه ریسکی کاهش پیدا کرد
- چه چیزی هنوز خارج از محدوده است

بدون گزارش، تغییر فنی از نظر مالی قابل دفاع نیست.
