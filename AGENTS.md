# DeDe — AI Working Rules

این فایل قوانین عملیاتی اجباری برای هر عامل هوش مصنوعی، به‌ویژه Codex، در پروژه DeDe است.

## 1. قبل از شروع هر کار

1. ابتدا `docs/PROJECT-MASTER.md` را کامل بخوان.
2. سپس فایل‌های واقعی مرتبط با همان تسک را بررسی کن.
3. اگر دستور کاربر با `PROJECT-MASTER.md` تعارض داشت، آخرین دستور صریح کاربر مقدم است و همان تغییر باید در `PROJECT-MASTER.md` ثبت شود.
4. اسناد داخل `docs/archive/` یا `tools/archive/` فقط تاریخی‌اند و مرجع جاری نیستند.
5. این آرشیوها باید در Git باقی بمانند؛ فقط فایل‌های فشرده، بکاپ و خروجی‌های generated خارج از Git هستند.

## 2. محدوده تغییر

- فقط همان تسک درخواست‌شده را انجام بده.
- تغییرات باید کوچک، متمرکز، قابل review و قابل revert باشند.
- branch جدید نساز؛ روی branch کاری فعلی ادامه بده.
- unrelated refactor، بازنویسی گسترده یا تغییر معماری بدون درخواست انجام نده.
- WordPress core، `wp-config.php`، uploads، cache، logs، backups، افزونه‌های عمومی/تجاری و فایل‌های محیطی را لمس نکن.
- فایل‌های generated، archive، `node_modules`، `vendor`، cache و log را وارد Git نکن؛ مگر dependency موجود پروژه عمداً تحت کنترل نسخه باشد و تسک صریحاً آن را لازم بداند.

## 3. فارسی و UTF-8

- تمام فایل‌ها را با UTF-8 سالم حفظ کن.
- متن‌های فارسی، label، placeholder، validation، button، option، پیام خطا و متن‌های تاریخ را بدون درخواست تغییر نده.
- هیچ متن فارسی نباید به `????`، mojibake یا کاراکتر خراب تبدیل شود.
- قبل از پایان کار، فایل‌های فارسی تغییرکرده را از نظر encoding بازبینی کن.

## 4. معماری و مالکیت قابلیت‌ها

- `DeDeV1` و `DeDeV2` فعلاً legacy هستند و حذف ناگهانی ممنوع است.
- انتقال قابلیت‌ها باید تدریجی، تست‌پذیر و بدون شکستن سایت باشد.
- افزونه‌های هدف:
  - `dede-core`
  - `dede-product-operations`
  - `dede-store-features`
  - قالب هدف آینده: `dede-theme`
- هر component جدید با نام `dede-*` باید در Git قابل track باشد و root آن به `tools/cpanel-deploy-allowlist.txt` اضافه شود.
- مالکیت دقیق هر قابلیت در `docs/PROJECT-MASTER.md` تعریف شده است.
- منطق تجاری جدید را داخل قالب قرار نده، مگر `PROJECT-MASTER.md` صریحاً آن را متعلق به قالب بداند.

## 5. افزونه یا قالب اختصاصی جدید

هر زمان component اختصاصی جدیدی ساخته شد:

1. root آن را به `tools/cpanel-deploy-allowlist.txt` اضافه کن.
2. مطمئن شو اسکریپت export دوره‌ای آن را از allowlist پوشش می‌دهد.
3. مسئولیت و مالکیت آن را در `docs/PROJECT-MASTER.md` ثبت کن.
4. به یاد داشته باش deploy شدن افزونه به معنی فعال شدن آن در وردپرس نیست.

## 6. Deploy و ایمنی

- مقصد فعلی deploy فقط `dede.ir` است.
- مسیر live: `/home/dedeir/public_html`
- مسیر repo روی cPanel: `/home/dedeir/repositories/dede`
- SSH در دسترس نیست؛ دستورهای SSH برای deploy پیشنهاد نده.
- روش اصلی deploy: cPanel Git Version Control.
- روند ثابت:
  1. بررسی تغییرات در GitHub Desktop
  2. commit و push
  3. cPanel: `Update from Remote`
  4. cPanel: `Deploy HEAD Commit`
  5. تست سریع سایت
- فقط فایل‌های تغییرکرده و مجاز deploy شوند.
- قبل از overwrite هر فایل live، بکاپ هدفمند همان فایل گرفته شود.
- حذف فایل در Git نباید خودکار فایل live را حذف کند، مگر بعداً سیاست صریح جدیدی تصویب شود.

## 7. مستندات

- `README.md` فقط گزارش قابل مشاهده کارفرما است.
- گزارش README باید کوتاه، واقعی، فارسی و مناسب فاکتور باشد.
- ساختار گزارش:
  - `## فاز X — عنوان`
  - `### هدف / اثر این فاز`
  - `### موارد انجام‌شده`
- در README ننویس:
  - وضعیت
  - مبلغ
  - «خروجی قابل ارائه»
  - متن تبلیغاتی یا اغراق‌آمیز
- تصمیم معماری، مسیر، فاز، scope یا قرارداد جدید باید در `docs/PROJECT-MASTER.md` ثبت شود.
- قوانین کاری Codex فقط در همین `AGENTS.md` نگهداری شوند.
- برای هر تسک چند فایل گزارش موازی ایجاد یا به‌روزرسانی نکن.

## 8. تست

- قبل از تغییر، مسیرهای تحت تأثیر را مشخص کن.
- حداقل syntax check مناسب فایل‌ها را انجام بده.
- تست دستی موردنیاز را صریح اعلام کن.
- برای تغییرات WooCommerce، اثر روی محصول، variation، cart، checkout، account و order را بررسی کن.
- چک‌لیست پایه موجود در `docs/LOCAL_BASELINE_TEST_CHECKLIST.md` را مرجع تست regression قرار بده.

## 9. پاسخ نهایی Codex

در پایان هر تسک فقط این موارد را گزارش کن:

- خلاصه کوتاه تغییر
- فایل‌های تغییرکرده
- تست‌ها یا syntax checkهای انجام‌شده
- تست دستی لازم
- ریسک یا نکته باقی‌مانده

از توضیح تبلیغاتی، کلی‌گویی و ادعای تستی که واقعاً انجام نشده خودداری کن.
